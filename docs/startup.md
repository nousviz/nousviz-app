# NousViz Startup Guide

Everything that happens when you run NousViz for the first time — what's automatic, what requires manual intervention, and what to do when something goes wrong.

---

## Overview

```
scripts/setup.sh          ← one-time setup (installs Postgres, deps, migrations, generates key)
    ↓
uvicorn apps.api.src.main:app  ← API server
    ↓
npm run dev               ← frontend dev server
    ↓
Open browser → Setup wizard  ← set password and enable auth (first run only)
```

---

## Step 1 — `scripts/setup.sh`

Run once after cloning. Safe to re-run (all steps are idempotent).

### What it does automatically

| Step | What happens |
|------|-------------|
| `.env` copy | Copies `.env.example` → `.env` if `.env` doesn't exist |
| Postgres install | Installs PostgreSQL natively via Homebrew (macOS), apt (Debian/Ubuntu), dnf (RHEL/Fedora), or pacman (Arch) if not already present |
| Postgres start | Starts the Postgres service via `brew services` or `systemctl` |
| DB + user creation | Creates the `nousviz` database and `nousviz` user (using the superuser to bootstrap) |
| Python venv | Creates `.venv` and installs `apps/api/requirements.txt` |
| Frontend deps | Runs `npm install` in `apps/web` |
| Migrations | Runs all `.sql` files in `storage/postgres/migrations/` in order |
| Plugin schemas | Runs `src/setup_schema.py` for any installed plugins |

### What requires manual intervention

**After setup completes**, start the API and open the browser. No manual `.env` editing is required for basic operation.

`NOUSVIZ_ENCRYPTION_KEY` is auto-generated by `setup.sh` and written to `.env` automatically.

`AUTH_REQUIRED` is set via the **Setup Wizard** on your first browser visit. The wizard creates the first superadmin (email + password). Subsequent users join via admin-issued invites — multi-user is the only auth model.

**No Docker required.** Postgres is installed and managed natively by your system.

## Step 2 — Environment variables

`.env` is created automatically by `setup.sh`. No manual editing is required for basic operation.

### Auto-configured by setup

`NOUSVIZ_ENCRYPTION_KEY` is auto-generated by `setup.sh` on first run. If it was not generated (e.g. Python was not available during setup), re-run `./scripts/setup.sh` or set it manually:
```bash
python3 -c "import secrets; print(secrets.token_hex(32))"
# Add to .env: NOUSVIZ_ENCRYPTION_KEY=<value>
```

### Configured via the setup wizard (first browser visit)

| Variable | Default | How it gets set |
|----------|---------|----------------|
| `AUTH_REQUIRED` | `false` | **Setup wizard** — Security step (enable auth checkbox) |
| First superadmin user (email + password) | *none* | **Setup wizard** — Security step writes the row directly to the `users` table |
| `POSTGRES_PASSWORD` | random, generated by `setup.sh` | **Settings → Database** to rotate |

`AUTH_REQUIRED` can be changed after setup via the Security step (re-accessible from Settings). The superadmin's email/password are managed in **System → Users** (or via [`scripts/reset-password.sh`](../scripts/reset-password.sh) if SMTP isn't configured and you forgot the password).

### Optional

| Variable | Default | Purpose |
|----------|---------|---------|
| `OPENROUTER_API_KEY` | *(blank)* | Enables AI chat in the dashboard |
| `API_PORT` | `8000` | Change if port is already in use |
| `WEB_PORT` | `5173` | Change if port is already in use |
| `CORS_ORIGINS` | `http://localhost:5173,...` | Add your production domain |

### ClickHouse — not needed for core

The core platform runs on PostgreSQL only. ClickHouse is an optional utility plugin (shipped in v0.2.1) for analytics plugins that need high-performance aggregation over large datasets. Install it from the Marketplace when needed — the utility plugin handles binary install, PM2 management, and config.

If you install the ClickHouse utility, its install script writes `CLICKHOUSE_HOST`, `CLICKHOUSE_PORT`, `CLICKHOUSE_USER`, `CLICKHOUSE_PASSWORD`, and `CLICKHOUSE_DATABASE` to `.env` automatically. You do not need to pre-configure these. To connect to an external ClickHouse instead of the bundled one, edit these values on the utility plugin's Settings page at `/plugin/clickhouse/settings`.

## Step 3 — API Server Startup

```bash
source .venv/bin/activate
python3 -m uvicorn apps.api.src.main:app --reload --port 8000
```

### What happens on startup

1. **CORS middleware** registers allowed origins from `CORS_ORIGINS`
2. **Auth middleware** registers — enforces session-token or API-key auth on every request when `AUTH_REQUIRED=true`
3. **Error sanitization middleware** registers — catches unhandled exceptions, returns safe messages
4. **Core routes** register (~20 route modules, always present regardless of plugins)
5. **Plugin loader** scans `plugins/installed/*/api/routes.py` and dynamically registers any installed plugin routers

### Startup log to expect (no plugins installed)

```
INFO     nousviz.plugin_loader: Loaded 0 plugin(s): []
INFO     uvicorn: Application startup complete.
```

ClickHouse is **not checked at startup**. It is only used if an installed plugin requires it, and only at request time.

### If a plugin fails to load

The API **does not crash** — the plugin is skipped and logged:
```
ERROR    nousviz.plugin_loader: Plugin my-plugin: failed to load routes — <reason>
```
All other plugins and core routes continue to work normally.

---

## Step 4 — Frontend Dev Server

```bash
cd apps/web
npm run dev
```

Starts Vite on `http://localhost:5173` (or `WEB_PORT` if set in `.env`).

Proxies all `/api/*` requests to the API server at `http://localhost:8000` (or `API_PORT` if set).

**The frontend will load even if the API is down** — pages will show loading/error states rather than a blank screen.

## Common Issues

### Postgres not running after setup
```
✗ Could not connect to Postgres on localhost:5432
```
`setup.sh` starts Postgres automatically. If it's not running after setup:
- macOS: `brew services start postgresql@16`
- Linux: `sudo systemctl start postgresql`
- Or check whether the service is running: `pg_isready -h localhost`

### Postgres migration fails
```
✗ 001_connections_and_credentials.sql FAILED — check DB credentials in .env
```
- Check `POSTGRES_HOST`, `POSTGRES_USER`, `POSTGRES_PASSWORD` in `.env`
- Verify Postgres is running and accessible: `pg_isready -h localhost -U nousviz`

### Migration crashes mid-run with `relation "credentials" does not exist`

This was a deterministic bug in every release v0.9.0 → v0.9.10.7 (ticket
B244). The pre-v0.9.11.0 migration runner ran `*_down.sql` files as
forward migrations; migration `046_credentials_restore_down.sql` dropped
the `credentials` table, then `047_nousviz_plugin_role.sql` crashed
trying to `REVOKE ALL ON credentials FROM nousviz_plugin`.

**Symptom**:
```
[6/7] Running Postgres migrations...
  ✓ 046_credentials_restore.sql
  ✓ 046_credentials_restore_down.sql       ← bug: ran the rollback
Traceback …
psycopg2.errors.UndefinedTable: relation "credentials" does not exist
CONTEXT:  SQL statement "REVOKE ALL ON credentials FROM nousviz_plugin"
```

**Re-running setup hits the same crash** — there's no recovery path
short of wiping the DB. The runner skips the `_down.sql` rows it
already "applied" but re-attempts 047, which still fails because
`credentials` was dropped.

**Fix**: upgrade to v0.9.11.0 or later (the runner now skips `_down.sql`
files), then re-run setup against a clean DB:

```bash
# 1. Wipe the partially-installed schema entirely.
sudo -u postgres dropdb nousviz
sudo -u postgres dropuser nousviz
sudo -u postgres dropuser nousviz_plugin
sudo -u postgres dropuser nousviz_query

# 2. Update to v0.9.11.0+.
git fetch && git checkout v0.9.11.0   # or any tag >= v0.9.11.0

# 3. Re-run setup. The fixed migration runner skips _down.sql files.
sudo ./scripts/setup.sh --server
```

**Production servers running v0.9.10.7 or earlier are NOT affected** —
they were provisioned before the bug surfaced and their
`schema_migrations` ledger is paid forward. Only fresh installs hit
this. See [storage/postgres/migrations/README.md](../storage/postgres/migrations/README.md)
for more.

### Port already in use
```
ERROR: [Errno 48] Address already in use
```
Set `API_PORT` or `WEB_PORT` in `.env` to a free port, then restart.

### `NOUSVIZ_ENCRYPTION_KEY` not set
Credential save/read operations will fail with a 500 error. Re-run `./scripts/setup.sh` — it will generate and write the key automatically. If Python is not available, generate manually:
```bash
python3 -c "import secrets; print(secrets.token_hex(32))"
```
Add `NOUSVIZ_ENCRYPTION_KEY=<value>` to `.env`, then restart the API.

### ClickHouse errors from a plugin
If you install an analytics plugin and see ClickHouse connection errors, the plugin's setup wizard will guide you through installing and configuring it. ClickHouse is not required for the core platform.

## Production / Server Deployment

Use `./scripts/setup.sh --server` on any Ubuntu/Debian server (DigitalOcean, Hetzner, AWS, etc.). This runs the same setup as local dev, plus:

- Builds the React frontend for production (`npm run build` → `apps/web/dist/`)
- Installs pm2 globally if not present
- Starts the API and alert worker via `ecosystem.config.js`
- Prints nginx setup commands and `.env` reminders

### Full server setup steps

```bash
# 1. Clone and setup
git clone https://github.com/nousviz/nousviz-app.git
cd nousviz
sudo ./scripts/setup.sh --server

# 2. Configure nginx to serve the React dashboard and proxy the API
sudo cp infra/nginx.conf /etc/nginx/sites-available/nousviz
sudo ln -sf /etc/nginx/sites-available/nousviz /etc/nginx/sites-enabled/nousviz
sudo rm -f /etc/nginx/sites-enabled/default
sudo nginx -t && sudo systemctl reload nginx

# 3. Open http://your-server-ip in a browser
#    The setup wizard will guide you through securing the instance:
#    - Create the first superadmin (email + name + password)
#    - Enable authentication (AUTH_REQUIRED=true)
#    Completing the wizard writes AUTH_REQUIRED to .env and inserts the
#    first user row directly. Takes effect immediately — no restart needed.
```

**`.env` values still requiring manual attention after setup:**

| Variable | What to set | How |
|----------|-------------|-----|
| `NOUSVIZ_ENCRYPTION_KEY` | Auto-generated by `setup.sh` | Nothing to do — already set |
| `CORS_ORIGINS` | Your domain (if using a custom domain) | `nano .env` then `pm2 restart api` |
| `POSTGRES_PASSWORD` | A strong password | Settings → Database after first login |

### Make pm2 survive reboots

```bash
pm2 startup   # prints a sudo command — paste it and run it
pm2 save
```

### Scheduled jobs run under PM2 (not system crontab)

All recurring work on a NousViz server — alert evaluation, system health snapshots, plugin data syncs — is scheduled via PM2's `cron_restart`, declared in `ecosystem.config.js`. There is **nothing to install in the system crontab** on a standard deploy.

You can see the live schedules via:

```bash
pm2 list                          # shows cron_restart jobs (alerts, health-monitor, etc.)
pm2 jlist                         # JSON output — Settings → Scheduled Jobs reads this
```

Or in the browser: Settings → Scheduled Jobs. The widget shows each job, its schedule, last-run timestamp, and whether PM2 is actively scheduling it.

**To change a schedule** (for example, make the alert runner fire every 15 minutes instead of every hour):

1. Edit `ecosystem.config.js` — find the job, change `cron_restart` to your new cron expression.
2. `pm2 reload ecosystem.config.js --update-env` — graceful reload, picks up the new schedule.
3. `pm2 save` — persists the new state so it survives a server reboot.

**To change a plugin sync schedule** without editing the ecosystem config: override it in the plugin's settings. Settings → Plugin → (plugin name) → Settings — the `sync_schedule` field accepts any cron expression, takes effect immediately. Reverts to the plugin manifest default when cleared.

**The install-crontab code block in Settings → Scheduled Jobs only appears on bare-metal deploys** (no PM2 detected). PM2-managed installs see a "Scheduled jobs run under PM2" note instead — because pasting the crontab block on a PM2 deploy would install duplicate jobs running alongside PM2's.

### Common server issues

**API returns 502 via nginx**
- Check `pm2 status` — `api` process must be `online`
- Check `pm2 logs api` for startup errors
- Confirm gunicorn is installed: `.venv/bin/gunicorn --version`

**Browser API requests blocked (CORS error)**
- `CORS_ORIGINS` in `.env` must include your exact domain (no trailing slash)
- Example: `CORS_ORIGINS=https://app.example.com`
- Restart after changing: `pm2 restart api`

**Frontend shows 404 for sub-routes (e.g. /marketplace)**
- `try_files $uri $uri/ /index.html` must be in your nginx config
- The provided `infra/nginx.conf` has this correctly — check you copied it

**`VITE_API_URL` — leave blank for server deploys**
- When using nginx as a reverse proxy, `VITE_API_URL` must be blank
- nginx routes `/api/*` to the backend automatically
- Setting it to `http://localhost:8000` will break browser requests from your domain

## HTTPS setup (server only — domain required)

**A real domain is required.** Self-signed certificates on a bare IP are not supported — browser security warnings make the platform effectively unusable. If you don't have a domain, register one before continuing. Any registrar works.

### Prerequisites

- DNS **A record** pointing your domain at the server's public IP. Verify with:
  ```bash
  dig +short app.example.com
  ```
  The output must match the server's IP.
- **Port 80 open** on the server (Let's Encrypt HTTP-01 challenge needs it). Check with `sudo ufw status` and your cloud provider's security group / firewall rules.
- **Port 443 open** on the server (for HTTPS itself).
- Nginx already installed and the site config in place — done automatically by `./scripts/setup.sh --server`.
- Root / sudo access.

### Run the setup script

```bash
sudo ./scripts/ssl-setup.sh app.example.com
```

Or run it with no arguments to be prompted interactively:

```bash
sudo ./scripts/ssl-setup.sh
```

### What the script does

1. **Verifies DNS** — resolves your domain and confirms it points at this server; warns if mismatched
2. **Installs certbot** if missing (apt on Debian/Ubuntu, brew on macOS)
3. **Prompts for a Let's Encrypt renewal email** — you'll get 20-day and 10-day expiry warnings here
4. **Obtains a certificate** via the HTTP-01 challenge (why port 80 needs to be open)
5. **Rewrites** `/etc/nginx/sites-available/nousviz` using the SSL template at `infra/nginx-ssl.conf`
6. **Reloads nginx**
7. **Updates `CORS_ORIGINS`** in `.env` to include `https://<your-domain>`
8. **Auto-renewal** is handled by certbot's system-installed systemd timer — no cron to configure

### Verification

```bash
# Health endpoint over HTTPS
curl -I https://app.example.com/api/health
# → HTTP/2 200

# Renewal dry-run — confirms auto-renew will work in 60 days
sudo certbot renew --dry-run
# → Congratulations, all renewals succeeded

# Systemd timer installed
systemctl list-timers | grep certbot
```

In the app, the topbar health dropdown should show `HTTPS · Let's Encrypt · app.example.com`.

### Behind a CDN (Cloudflare, CloudFront, Fastly, Netlify)?

If your domain is proxied through a CDN, Let's Encrypt HTTP-01 can't reach your origin — the CDN intercepts the request. You have two paths:

**Option A — Use the CDN's edge SSL (recommended for most OSS operators).**

Most CDNs serve HTTPS at the edge for free. Skip this wizard entirely:

- **Cloudflare**: SSL/TLS → set mode to `Flexible` (origin stays HTTP) or `Full (strict)` (origin also has a cert). Cloudflare issues and renews the edge cert automatically. See [Cloudflare SSL docs](https://developers.cloudflare.com/ssl/).
- **AWS CloudFront**: Request a cert in AWS Certificate Manager, attach it to the distribution. ACM handles renewal.
- **Fastly**: Enable TLS in your Fastly account. Fastly provisions the cert.
- **Netlify**: Automatic. Add the custom domain in site settings; Netlify issues Let's Encrypt certs and renews them for you.

Your origin server stays on HTTP internally. The CDN terminates HTTPS at the edge. No `ssl-setup.sh` needed.

**Option B — Let's Encrypt at the origin (end-to-end encryption).**

If you want the connection between the CDN and your server to be encrypted too:

1. Temporarily disable the CDN proxy for this domain:
   - **Cloudflare**: click the orange cloud on the A record — it turns grey (DNS-only)
   - Other CDNs: consult their docs for "bypass proxy" / "direct origin" mode
2. Wait ~1 minute for DNS to repropagate.
3. Run `sudo ./scripts/ssl-setup.sh <domain>` — Let's Encrypt will issue a cert at the origin.
4. Re-enable the CDN proxy.
5. **Cloudflare**: set SSL/TLS mode to `Full (strict)` so Cloudflare trusts your origin cert.

The wizard now detects common CDN IP ranges and guides you through the same two options if you try to run it against a proxied domain.

### Common HTTPS issues

**Certbot fails with "DNS problem: NXDOMAIN"**
- DNS hasn't propagated yet. Check `dig +short <domain>` returns your server IP.
- Wait up to an hour after creating a new A record; check again.

**Certbot fails with "Fetching http://... : Timeout during connect"**
- Port 80 is blocked. Check `sudo ufw status`, your cloud provider's security group, and any upstream firewall.
- Cloudflare proxying can also block this if orange-cloud is enabled — disable the proxy for the first cert issuance, enable after.

**After switch, the app returns CORS errors**
- Check `.env`: `CORS_ORIGINS` should include `https://<domain>` (with `https`, not `http`). The script does this automatically; verify if you see CORS errors in the browser console.

**Renewal fails 60 days later**
- Run `sudo certbot renew --dry-run` — if it fails, the systemd timer may be broken. `systemctl status certbot.timer` to inspect.
- Usually resolved by `sudo apt install --reinstall certbot` on Debian/Ubuntu.

**Browser shows "your connection is not private"**
- You're on an IP instead of the domain — go to `https://<domain>` not `https://<ip>`.
- Or the cert is expired — check `curl -I https://<domain>/api/health` and the server's certbot state.

**`http://<server-ip>/` returns 404 after setup**
- Expected if you're running a version before v0.3.0. After certbot reshapes nginx, only `<domain>` has a matching server block. From v0.3.0 onward the wizard installs a default-server block that redirects any bare-IP access to `https://<domain>/`.
- If you need to restore bare-IP access on an older install, add `listen 80 default_server; server_name _; return 301 https://<domain>$request_uri;` to `/etc/nginx/sites-available/nousviz-bare-ip-redirect` and symlink it into `sites-enabled`.

**PM2 processes disappear after SSL setup (pre-v0.3.0)**
- Older versions of `ssl-setup.sh` used `pm2 kill && pm2 start ecosystem.config.js`, which wiped every PM2 process including utilities started separately. Fixed in v0.3.0 by switching to `pm2 reload all --update-env`.
- Recover with `pm2 resurrect` to restore from the last saved state.
